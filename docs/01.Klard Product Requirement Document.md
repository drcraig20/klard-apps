### 5.4 BurnerCard System (â€œKlard Pay-Onceâ€)

The **BurnerCard** is the heart of Klardâ€™s financial control layer.

**Primary Purpose:**

Enable users to create **virtual payment cards** tied to subscription rules and automatically block any unauthorized renewals or price hikes.

**Core Workflow:**

1. User requests a new BurnerCard via the dashboard.
2. Klard backend (Spring service) invokes an issuing API (e.g., **Lithic**, or **Airwallex Issuing**).
3. Card metadata (masked PAN, expiry, limits) returned and securely stored.
4. User uses the card to pay for a subscription.
5. Klard monitors charge events via webhooks.
6. Based on user-defined rules, the system **auto-locks**, **limits**, or **deletes** the card.

**Rule Configuration Options:**

| Rule | Description |
| --- | --- |
| **Card Type** | One-time (single transaction), Recurring-limited, or Category-locked. |
| **Linked Subscription** | Card can be tied to a single `subscription_id`. |
| **Linked Category** | Limit use to specific category (e.g., â€œStreamingâ€ or â€œProductivityâ€). |
| **Max Recurring Count** | For recurring cards, user can specify **max N charges** before **auto-lock**. |
| **Spending Limit** | Hard cap (e.g., $20 total). |
| **Expiry Rule** | Auto-lock after **X days** even if unused. |
| **Notification Rule** | Alert user when 80 % of limit reached or when next charge attempt occurs. |

**System Behaviors:**

- Auto-block after rule breach.
- Decline unauthorized merchant attempts.
- Mark subscription as â€œBlockedâ€ or â€œCompleted.â€
- Log savings event in dashboard: *â€œ$14.99 charge prevented â€” card expired.â€*
